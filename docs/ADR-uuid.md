# ADR: UUID IDs for Plans, Groups, and Places

## Status
Accepted - Implementation pending

## Context
Currently, IDs for plans are generated using `plan_` + timestamp (`'plan_' + Date.now()`). Groups and places use `crypto.randomUUID()` but the plan ID generation is inconsistent and not truly UUID-based.

## Decision
All IDs should be UUIDs generated by `crypto.randomUUID()` for consistency and uniqueness guarantees.

### ID Types

| Entity | Current ID Format | New ID Format |
|--------|-------------------|---------------|
| Plan | `plan_` + timestamp (e.g., `plan_1718956789012`) | UUID (e.g., `a1b2c3d4-e5f6-7890-abcd-ef1234567890`) |
| Group | UUID via `crypto.randomUUID()` | No change (already correct) |
| Place | UUID via `crypto.randomUUID()` | No change (already correct) |

### Benefits of UUIDs

1. **Global uniqueness**: Cryptographically guaranteed unique across space and time
2. **No collisions**: Even with rapid creation or offline scenarios
3. **Consistency**: All entities use the same ID generation method
4. **Security**: No information leakage (timestamp IDs reveal creation order/timing)

## Implementation Steps

1. Update `createPlan()` to use `crypto.randomUUID()` for plan ID
2. Update `importPlan()` to generate UUID for imported plans if missing
3. Update migration logic in `loadFromLocalStorage()` to ensure plan has UUID
4. Update export functions to include plan ID

## Data Structure Changes

### Plan Object (Before)
```js
{
  id: "plan_1718956789012",  // timestamp-based
  name: "東京之旅",
  groups: [...],
  _meta: { ... }
}
```

### Plan Object (After)
```js
{
  id: "a1b2c3d4-e5f6-7890-abcd-ef1234567890",  // UUID
  name: "東京之旅",
  groups: [...],
  _meta: { ... }
}
```

## Consequences

- **Breaking**: Old plan IDs will change on migration (acceptable for v1)
- **Storage key unchanged**: `travel-map-plans` remains the same
- **Backward compatible**: Migration can convert old timestamp IDs to UUIDs
